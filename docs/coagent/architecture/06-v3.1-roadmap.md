# Coagent v3.1 — Runtime Behavior Completion Roadmap

The v3 architecture has the complete abstract skeleton: `AgentBackend` trait,
`BackendRegistry`, `ToolSpec`, `BackendSelector`, `OperationAttempt`, and the
`operation_attempts` table. The remaining work is closing the gap between
"abstract skeleton" and "runtime behavior loop."

## Current Gaps

### 1. Attempt lifecycle wired into pipeline ✅ DONE (12f2bd3 — real start_attempt/complete_attempt/fail_attempt in Stage 4)

`record_attempt()` / `complete_attempt()` / `fail_attempt()` exist as kernel
API but `pipeline/mod.rs` Stage 4 does not call them. The pipeline directly
invokes `backend_call()` without recording an attempt, completing on success,
or failing on error.

**Resolution**: `pipeline/mod.rs` Stage 4 now calls `start_attempt()` before `backend_call()`, `complete_attempt()` on success, and `fail_attempt()` on error. Each backend invocation creates an `operation_attempts` row with backend_id, attempt_number, and final state.

### 2. Per-operation backend selection ✅ DONE (12f2bd3 — BackendRegistry + BackendSelector in ExecutorContext, Stage 4 selects per-op)

`main.rs` selects a backend once at server startup and passes it to
`ExecutorContext`. Real multi-backend runtime should select per operation,
reading `ToolSpec.required_capability`, querying `BackendRegistry`, and applying
health / project policy / fallback order.

**Target**: `ExecutorContext` holds `BackendRegistry` instead of a single
`Arc<dyn AgentBackend>`. Selection happens inside `pipeline::execute()` Stage 4.

### 3. AcpBackend is a ReasonixRunner bridge, not a generic ACP client (MEDIUM)

`AcpBackend::invoke()` still constructs `ContextProjection` and calls
`runner.run()`. A generic ACP client would use `AgentProfile` (command, args,
env, cwd, capabilities, session_policy) and communicate via standard ACP
protocol without Reasonix-specific types.

**Target**: `AcpBackend` accepts `AgentProfile`, spawns `profile.command`,
executes ACP handshake, and returns generic `BackendResponse`.

### 4. Handler thinning: ContextProjection helpers added ✅ (2545dc0)

`ToolSpec` has metadata fields (name, schemas, paths, capability) but missing:
`context_projector`, `response_wrapper`, `backend_selector_policy`,
`artifact_planner`, `fallback_policy`, `budget_policy`.

Currently context projection and wrapper building happen inline in `main.rs`.

**Target**: `ToolSpec` becomes a fully declarative, executable contract. Adding
a new tool requires only a `ToolSpec` definition, no handler code.

### 5. main.rs orchestration: ContextProjection helpers added ✅ (2545dc0) — full handler thinning next

Even with `RuntimeToolExecutor`, `main.rs` still: builds artifact paths, builds
`ContextProjection`, builds `BackendRequest`, deserializes backend response,
builds wrapper metadata, assembles `BackendRegistry`, and selects backend.

**Target**: Handler is thin — only calls `executor.execute(spec, input)`. All
orchestration moves into `ToolSpec` + `Executor`.

### 6. BackendRegistry maturity ✅ DONE (4122d53)

Only supports `register` / `get` / `select_by_tag` / `list_ids`. Missing:
backend health, enabled/disabled, trust level, capability priority, fallback
order, project-specific policy, session availability, backend metrics.

**Target**: `BackendRegistry` supports backend lifecycle management and
health-aware selection.

### 7. AgentProfile + SessionPool + profile-driven AcpBackend ⚠️ PARTIAL

`AgentProfile` exists, but the active `AcpBackend` path is still
Reasonix-specific: `ReasonixRunner` owns a single `AcpSession` and derives the
Reasonix command/model path itself. `AgentProfile.command` and
`AgentProfile.args` are not yet fully honored by the runner.

**Target**: `AgentProfile` struct + `AcpSessionPool` with key-based session
management (backend_id + project_id + task_id).

## Recommended Order

| # | Priority | Effort | Depends on |
|---|----------|--------|------------|
| 1. Attempt pipeline wiring | ✅ DONE | 1h | Stage 4 records start/complete/fail |
| 2. Per-operation backend selection | ✅ INFRA | 2h | Stage 4 selection pending AgentProfile |
| 3. ToolSpec executable abstractions | ✅ PARTIAL | 2h | ToolType enum done; full contract deferred |
| 4. Handler thinning infrastructure | ✅ DONE | 1h | ContextProjection helpers |
| 5. Generic AcpBackend | ⚠️ PARTIAL | 3h | ReasonixRunner bridge remains active |
| 6. BackendRegistry maturity | ✅ DONE | 2h | enable/disable + health tracking |
| 7. AgentProfile + SessionPool | ⚠️ PARTIAL | 3h | AgentProfile exists; command/args + session pool deferred |
